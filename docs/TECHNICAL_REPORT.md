> інженерні рішення, валідація, обмеження. Для презентації/звіту/онбордингу.
> Поглиблені специфікації по підсистемах — `docs/architecture/` (14 файлів).
> Версія: 1.16.2 (+ audit-fix-pass). Дата: 2026-06-01.

---

## 1. Анотація
Mask Picker — система **ручної доразмітки** (human-in-the-loop) масок клітинної
мікроскопії. ML-моделі дають чорнові instance-маски; інструмент дозволяє анотатору
швидко відібрати/виправити/згрупувати їх у браузері та запекти **узгоджений
ground-truth датасет** (instance + semantic маски + групи «1 ядро + N везикул») для
downstream-алгоритму привʼязки везикул (clusterization). Ключова інженерна цінність —
**гарантована узгодженість даних** між трьома редакторами, двома ID-просторами і
файлами на диску.

## 2. Проблема й контекст
Автоматична сегментація (Cellpose/InstanSeg/StarDist/YOLO) на складних знімках дає
неідеальні маски: пропуски, хибні обʼєкти, злиплі клітини. Для навчання/валідації
алгоритму кластеризації потрібен **чистий розмічений датасет**. Ручна розмітка з нуля
надто дорога → потрібен інструмент, що бере найкращий ML-вихід і дає людині ефективно
його **доводити до ідеалу** та структурувати (групувати в клітини).

## 3. Огляд системи
**Стек:** Python/Flask (backend), vanilla ES-модулі + Canvas/SVG (frontend), numpy/
OpenCV/PIL (обробка масок), cellsegkit (рендер/експорт, запозичено з Cryobiology III).

**Три інструменти в одному редакторі (ітеративні, не конвеєр):**
1. **Cleanup** — відхилити погані маски (reject) + позначити пропуски (markers).
2. **Polygons** — домалювати/виправити маски полігонами (draw/pick/seed).
3. **Groups** — обʼєднати інстанси+полігони у клітини, класифікувати за правилами.

**Пайплайн даних:**
```
ML-сегментація → output/<model>/npy (raw, RO) → [Cleanup/Polygons/Groups у браузері]
   → autosave JSON на диск → BAKE (raw − rejected + polygons) → selected/npy (baked)
   → Finalize/--pack → dense 1..N + semantic + groups → датасет замовнику
```

## 4. Модель даних і ID-простори (серце системи)
Деталі → `architecture/01`. Стисло — **три простори instance-ID**:
- **raw_iid** — ID у виході моделі (`output/`, стабільний, read-only).
- **baked_iid** — ID у запеченій масці (`selected/`); уцілілі = raw; полігони = `50000+idx` (**reserved range**).
- **polygon_index** — індекс полігону в LabelMe-файлі.

**Reserved-ID range (50000+)** — фундаментальне рішення: полігони ніколи не конфліктують
з ID моделі, ID стабільні між запіканнями → групи не «дрейфують», а фронт може рахувати
ID локально (== фінальному bake). На це спирається все інтерактивне групування.

**Дві «різні памʼяті»:** ефемерний браузерний стан (поки відкритий редактор) ↔
персистентні файли-SoT на диску; синхронізація — per-domain autosave (debounce) + flush
на закритті + анти-race guard.

## 5. Bake-пайплайн (deliverable)
`raw → (− rejected=0) → (+ polygons у reserved ID) → backstop (стерти інстанси >50%
перекриті полігоном) → export (npy/png/yolo/overlay) → Layer-2 sync груп → strip-orphan`.
Для замовника — **compaction** у щільні `1..N` + `semantic` (per-pixel клас) + опц.
`mask_groups`. Деталі → `architecture/09`, `12`.

## 6. Ключові інженерні рішення
| Виклик | Рішення |
|---|---|
| Колізії ID (полігон ↔ відхилена клітина) | **Reserved-ID range 50000+** (v1.16.0) |
| Подвійний підрахунок / «привид» членів групи | **Layer 1/Layer 2** розмежування (count ігнорує bake-артефакти) |
| 7-сек фриз на кожну дію групування | **Solution B** — client-side hit-test з робочих даних (reserved-ID робить це коректним) |
| «Фікс тут — зламав там» (мульти-представлення) | **Концепт-орієнтований аудит** (`CODE_AUDIT_PRINCIPLES.md`) + системний прохід |
| Узгодженість undo на 3 домени | **Єдиний хронологічний стек** (складені дії = 1 запис) |
| Race autosave ↔ undo | **Generation-guard** + проміс-черга збережень |
| Огризки клітин у deliverable (50–85% перекриті) | **backstop вирівняно до 0.5** = UI covered (audit-fix F-011) |

## 7. Валідація
- **237 автотестів** (pytest) — bake, groups, data_sync, cleanup, polygons, інваріанти, регресії багів.
- **Browser-верифікація** ключових жестів (preview MCP — реальні пікселі/DOM, не «на око»).
- **Інваріант-верифікатори** (`audit_export.py`, `desync_invariants.py`, `verify_bug3_clusterization.py`).
- **Системний аудит** усіх multi-representation концептів (`architecture/14`) — розбіжностей у коді не виявлено.
- **End-to-end** перевірка deliverable на реальних даних (read-only) — структура й маски коректні.

## 8. Обмеження (відомі)
- Reserved-ID має стелю (датасети з >50k інстансів / >15k полігонів/фото → fallback зі старими ризиками; `F-013`).
- Semantic-клас кодується порядком `labels.json` (контракт із консюмером; `F-014`).
- `--pack` layout (`masks_npy/`) ≠ референс-скрипт замовника (`npy/`) — потребує узгодження імен (`F-009`).
- Multi-file bake не транзакційний (кожен файл атомарний; crash між writes малоймовірний).
